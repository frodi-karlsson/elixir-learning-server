Kafka is powerful, but it's not exactly known for its developer ergonomics. Recently, while building the @anabranch/eventlog-kafka adapter, I ran into a classic KafkaJS "gotcha" that felt like fighting a ghost: a mysterious, consistent 5-second delay on manual offset commits.

The @anabranch/eventlog abstraction allows for at-least-once delivery by letting you manually commit cursors (offsets) after processing a batch. When using the Kafka adapter, this eventually maps down to the KafkaJS admin.setOffsets call. And that's where the trouble started.

The 5-Second Mystery

In integration tests, every manual commit was taking exactly 5 seconds. If I did two commits, it took 10 seconds. It was maddeningly consistent. My initial thought was that the consumer group was stuck in a rebalance, but the delay happened even when the consumer was already disconnected.

The smoking gun appeared when digging into the KafkaJS source. It turns out that admin.setOffsets isn't just a simple metadata update. Under the hood, KafkaJS implements this by spinning up a "transitive" consumer. It joins the group, fetches the current state, sets the offsets, and leaves. And here's the kicker: it uses the default maxWaitTimeInMs of 5000ms for its internal fetch requests.

// Inside kafkajs/src/admin/index.js
const consumer = this.cluster.consumer({
  groupId,
  maxWaitTimeInMs: this.maxWaitTimeInMs, // Defaults to 5000!
  // ...
})

If the group is empty or the topic is idle, that internal consumer will sit there and wait for the full 5 seconds before giving up on the fetch and proceeding with the offset update. It's a safeguard built for high-throughput production environments (like LinkedIn's interactions), but it's a productivity killer for developers running local tests or low-latency administrative tasks.

The Anabranch Workaround

To fix this in the @anabranch/eventlog-kafka adapter, I had to bypass the high-level admin.setOffsets and implement a more surgical "out-of-band" commit using a temporary dummy consumer with a much lower timeout.

By manually controlling the lifecycle of this temporary consumer and setting maxWaitTimeInMs to something sane (like 100ms), we can jump into the group, seek to the target offsets, commit, and get out immediately.

async commitCursor(topic, consumerGroup, cursor) {
  // ... (transform cursor to offsetsToCommit)

  // We spin up our own dummy consumer to bypass the 5s admin default
  const dummyConsumer = state.kafka.consumer({
    ...options.consumer,
    maxWaitTimeInMs: options.admin?.dummyMaxWaitTimeInMs ?? 100,
    groupId: consumerGroup,
  });

  try {
    await dummyConsumer.connect();
    await dummyConsumer.subscribe({ topic, fromBeginning: true });

    await new Promise<void>((resolve, reject) => {
      // Once we get a FETCH event (or a timeout), we stop and resolve
      dummyConsumer.on(dummyConsumer.events.FETCH, () => {
        dummyConsumer.stop().then(resolve).catch(reject);
      });

      dummyConsumer.run({ eachBatchAutoResolve: false }).catch(reject);
      dummyConsumer.pause([{ topic }]);

      for (const { partition, offset } of offsetsToCommit) {
        dummyConsumer.seek({ topic, partition, offset });
      }
    });
  } finally {
    await dummyConsumer.disconnect();
  }
}

This implementation effectively does what admin.setOffsets *should* do for administrative tasks: update the pointer and leave without waiting for data that might not exist.

Why is KafkaJS like this?

KafkaJS is an incredible feat of engineering—a pure JavaScript client that handles the complex Kafka protocol without native dependencies. But its design often prioritizes safety and "LinkedIn-scale" stability over developer ergonomics. Many of its internal timeouts and retry strategies are hardcoded or buried deep in the configuration.

Wrestling with these primitives is part of the "Kafka tax." But once you wrap them in a cleaner abstraction like Anabranch's Stream and Task, you can hide that complexity from the rest of your application. The EventLog interface doesn't care about maxWaitTimeInMs; it just knows that when you call commit(), the cursor is saved.

The source for the Kafka adapter is on GitHub. If you're using Kafka with TypeScript, it might save you a few 5-second headaches.
